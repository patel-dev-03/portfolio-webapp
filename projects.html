Decentralize Network

Peer to peer Distributed Cloud network

  • Java, Unix
  • Socket Programming
  • Chord Protocol

An application authored in Java, it is purely decentralized cloud network. Using Chord Protocol- Scalable Peer-to-peer Lookup Service for Internet Applications. Used sockets to connect to the network ring. Each node of the CHORD architecture is aware of it's successor and predecessor.

First of all any node joins the CHORD network simply by calculating an ID on the basis of it's ip and port number then any new node coming in the ring joins the ring by communicating with any of the node in the ring and finding the its successor and hence it's position in the ring. Two types of peer pointer successor(predecessor) and finger table

Some of the challenges I solved during development:

  • The Single biggest reason for implementing Chord Protocol was to reduce the search in finger table. Reducing time from linear search for ith entry of node n for successor to (( n + 2i-1 ) mod m).
  • Chord needs to deal with nodes joining the system concurrently and with nodes that fail or leave voluntarily. A basic stabilization protocol is used to keep nodes' successor pointers up to date, which is sufficient to guarantee correctness of lookups. Those successor pointers are then used to verify and correct finger table entries, which allows these lookups to be fast as well as correct. finger table entries involved in the lookup are reasonably current, and the lookup finds the correct successor in O(log N) steps.
  • Free Riding and Efficiency - Avoiding free riding by compulsory sharing of resources before using resources on the network. By limiting new users to very limited network access .Executing file can be divided into multiple threads and threads can be distributed among all the available resources.

Capstone Project

Automated DevOps Pipeline for website production

  • Jenkins, CI/CD pipeline
  • AWS, EC2
  • Terraform, Docker
  • Git, GitHub, Webhook
  • Puppet, Selenium

A website production environment for successful production, testing, deployment of website on apache server in EC2. A git commit is picked up my webhook which start the pipeline. A new docker image is build for the production of website. It is tested using automated testing tool selenium. On successful testing the website will be push to deployment if its in main branch

I am particularly pleased with the choice I made to provision all of my infrastructure using Terraform. This was my first time using it, so I spent several hours learning the basics and applying that knowledge to my configuration. Terraform seems like a great tool to use in my pipelines going forward, especially if/as I branch out to other cloud providers.

Numerous challenges presented themselves throughout the development process. Some examples:

  • After working out the docker script logic locally, I used multiple AWS accounts for testing and production. Each account's resources were deployed from a separate repository, which made for a lot of copy/paste. This process was streamlined using github webhook feature.
  • An IAM permission scoping issue threw me for a loop, sidetracking the project for several hours as I erroneously refactored the Terraform configuration.
  • There were several key parts to this challenge that I had to (or wanted to) learn from scratch: Maven, Puppet, Terraform; and unit testing in Selenium. I have to write java code using seleium libraries and upload the jar file to the testing server.Had these topics been familiar to me already, I suspect this project would have taken far less time.

Future improvements may include: refactoring the Terraform configuration file; writing more comprehensive unit tests.

    • Python, Keras,
    • Convulation Neural Network
